# Documentation Audit

## Scope

This audit classifies every Markdown file currently stored in the repository as of 2026-03-14.

Assumptions used in this audit:

- First-party documentation includes repository-root Markdown files, `frontend/README.md`, and the current flat `docs/` tree.
- Vendored Markdown files are still documentation, but they are classified separately because they are upstream third-party material rather than project-owned docs.
- Planning/task files are classified so they can be preserved or retired intentionally instead of remaining undocumented exceptions.

## Audit Summary

- Audited first-party Markdown files: 30
- Audited third-party vendor Markdown files: 5
- First-party documentation is currently mixed across technical architecture, deployment notes, task logs, and planning files.
- The current `docs/` tree is flat and mixes developer, operator, and future user-facing concerns.
- Several first-party documents are German or mixed-language and must be translated, merged, or deprecated to comply with the English-only rule.
- Central navigation indexes do not exist yet for `docs/`, `docs/technical/`, or `docs/functional/`.

## Duplication and Placement Findings

- Architecture content is fragmented across `docs/architecture.md`, `docs/backend.md`, `docs/frontend.md`, and `docs/domain-model.md`.
- Eventing content is fragmented across `docs/eventing-analysis.md`, `docs/event-model.md`, and `docs/polling-fallback.md`.
- Deployment and operator guidance is fragmented across `docs/deployment.md`, `docs/backend-config.md`, `docs/headless-setup.md`, `docs/daemon-headless.md`, and `docs/runtime-validation.md`.
- Planning and roadmap material is split across `Tasks.md`, `More_Tasks.md`, and `docs/future-extensions.md`.
- The current flat `docs/` tree hides audience boundaries because developer, operator, and future end-user topics live side by side.

## Migration Priorities

1. Create the documentation indexes and style guide so later moves have a stable destination and naming convention.
2. Translate, merge, or deprecate the German and mixed-language first-party files before expanding feature documentation.
3. Split the current flat `docs/` content into technical architecture, deployment, testing, and functional roadmap tracks.
4. Remove or explicitly exclude non-English vendor duplicates if the repository is meant to satisfy a strict English-only documentation policy.
